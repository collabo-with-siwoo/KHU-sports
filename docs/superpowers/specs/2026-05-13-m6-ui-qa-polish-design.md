# M6 UI QA 폴리시 설계

## 목표

M6는 M0부터 M5까지 구현한 기능을 베타 공개 후보 수준으로 다듬는 단계입니다. 새 기능을 크게 추가하기보다, 이미 구현된 사용자 흐름과 운영자 흐름이 모바일, 데스크톱, 접근성, 로딩 상태, 오류 상태에서 안정적으로 보이도록 정리합니다.

핵심 목표는 기존 흐름이 “동작한다”에서 “사용자가 안심하고 쓸 수 있다”로 올라가게 만드는 것입니다.

## 범위

M6-A는 핵심 사용자와 운영자 플로우를 대상으로 합니다.

- 공개 화면: `/`, `/notices`, `/notices/[id]`, `/results`, `/results/[tournamentId]`
- 인증 및 회원 화면: `/login`, `/signup`, `/mypage`, `/mypage/scores`, `/mypage/score-results`
- 관리자 화면: `/admin`, `/admin/notices`, `/admin/notices/new`, `/admin/members`, `/admin/members/[userId]`, `/admin/tournaments`, `/admin/scores`

우선 기준은 모바일 375px 화면입니다. 동시에 결과표와 관리자 화면처럼 정보 밀도가 높은 화면은 데스크톱에서도 함께 확인합니다. 기존 Stitch 기반의 차분한 시각 방향과 운영 도구다운 밀도는 유지합니다.

## 제외 범위

M6-A에서는 다음 작업을 하지 않습니다.

- 참가 신청 전체 워크플로우 신설
- Tiptap 에디터 도입
- rate limit 또는 이메일 자동화
- 배치 작업
- 결제 기능
- 새 데이터 모델 추가
- 대규모 디자인 시스템 재작성

정책 결정이나 데이터베이스 마이그레이션이 필요한 문제는 이번 폴리시 작업에 섞지 않고 후속 작업으로 기록합니다.

## QA 패스

첫 단계는 로컬 앱을 실행한 뒤 대상 라우트를 데스크톱과 모바일 폭에서 확인하는 것입니다. 각 화면에서는 실제로 수정할 가치가 있는 문제만 기록합니다.

확인 항목은 다음입니다.

- 레이아웃 깨짐
- 텍스트 넘침 또는 긴 한국어 제목 줄바꿈 문제
- 버튼, 필터, 입력 필드 사용성 문제
- 접근 가능한 label 또는 이름이 없는 폼 컨트롤
- 안내가 부족한 빈 상태와 오류 상태
- 이유를 알 수 없는 비활성 버튼
- 느리게 느껴지는 로딩 패턴
- 다음 행동을 찾기 어려운 내비게이션 흐름

공개 화면은 선수, 학부모, 외부 방문자가 빠르게 이해할 수 있어야 합니다. 결과 화면은 개인정보 노출 경계를 유지하면서 스캔하기 쉬워야 합니다. 마이페이지는 M5에서 정리한 것처럼 가볍게 유지하고, 전체 기록을 다시 eager loading하지 않습니다. 관리자 화면은 화려한 홍보형 UI보다 반복 작업에 적합한 조용하고 밀도 있는 구성을 유지합니다.

## 수정 원칙

수정은 작고 국소적으로 진행합니다. 동일한 문제가 여러 화면에서 반복될 때만 공통 CSS나 공용 패턴을 손봅니다. 새 추상화보다 기존 CSS, 페이지 구조, 폼 상태, 안내 문구를 개선하는 쪽을 우선합니다.

다음은 피합니다.

- 장식 중심의 리디자인
- 운영 화면의 과한 히어로 구성
- 카드 안에 카드를 넣는 구조
- 전체 색상 체계를 흔드는 변경
- 기능 범위를 넓히는 리팩터

반응형 개선은 안정적인 폭, 줄바꿈, overflow 방지, 필요한 경우 테이블의 모바일 카드 대체를 중심으로 처리합니다. 375px 모바일에서도 버튼, 입력 필드, 파일 업로드, 필터, 페이지네이션이 눌리고 읽혀야 합니다.

## 접근성과 UX

기본 접근성은 이번 단계에서 반드시 확인합니다.

- 폼 컨트롤은 label 또는 명확한 accessible name을 가져야 합니다.
- 삭제, 탈퇴, 확정 같은 위험한 액션은 확인 문구가 분명해야 합니다.
- 키보드 포커스가 보이지 않게 사라지면 안 됩니다.
- 비활성 상태는 사용자가 왜 사용할 수 없는지 알 수 있어야 합니다.
- 빈 상태와 오류 상태는 다음 행동을 안내해야 합니다.
- 공개 화면의 오류나 빈 상태는 개인정보를 노출하지 않아야 합니다.

## 성능

프로젝트 기준은 마이페이지와 주요 공개 화면이 5초 이상 멈춘 것처럼 느껴지지 않는 것입니다. M6-A는 `/mypage`나 공개 인덱스 화면에 새로운 eager join을 추가하지 않습니다.

데이터가 많은 화면은 기존 페이지네이션과 제한된 조회를 유지합니다. 더 많은 데이터를 한 번에 가져오는 대신, 명확한 빈 상태, 필터, 페이지 이동, 로딩 안내를 우선합니다.

## 검증

M6-A 완료 전에는 다음 명령을 실행합니다.

- `npm run typecheck`
- `npm run lint`
- `npm run prisma:validate`
- `npm test`
- `npm run build`

브라우저 기반 확인은 대상 라우트 목록을 중심으로 진행합니다. 최소한 모바일 1개 폭과 데스크톱 1개 폭에서 핵심 화면을 확인합니다. 최종 보고에는 실제 수정한 문제와 후속으로 미룬 문제를 구분해서 적습니다.

## 문서화

M6-A가 화면 동작, 라우트 기대값, 사용자 흐름을 바꾸면 다음 문서를 함께 갱신합니다.

- `docs/specs.md`
- `docs/spec-changelog.md`
- `docs/context.md`

순수 CSS 폴리시처럼 서비스 계약을 바꾸지 않는 변경은 최종 응답에서 요약하는 것으로 충분합니다.
